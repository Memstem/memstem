# ADR 0022: Codex adapter

Date: 2026-05-16
Status: Accepted

## Context

Memstem ingests memories, sessions, and skills from each AI client that
drops files on disk. Adapters exist for Claude Code and OpenClaw / Ari.
OpenAI's Codex CLI is the next first-class client: it stores rich
session logs and a growing user-skills directory, and ‚Äî as of the
0.130 series ‚Äî the Codex desktop app can launch the remote `codex`
binary over SSH so the same on-disk artifacts now also accumulate
during remote-development sessions.

A user running Codex on the same host as the Memstem daemon should
have those sessions and skills show up in unified search without any
additional cooperation from Codex itself ‚Äî the same property that
makes the Claude Code and OpenClaw adapters durable across vendor
upgrades.

Inspecting `~/.codex/` on a live install yields four ingest-relevant
locations:

| Path | Contents | Decision |
|---|---|---|
| `~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl` | One JSONL per session, partitioned by date | Ingest as `type: session` |
| `~/.codex/skills/<name>/SKILL.md` | User-authored skills (markdown + frontmatter) | Ingest as `type: skill` |
| `~/.codex/skills/.system/<name>/SKILL.md` | OpenAI-shipped skills (`imagegen`, `skill-creator`, etc.) | **Skip** ‚Äî vendor content, not user knowledge |
| `~/.codex/memories/*.md` | User memory files | Ingest as `type: memory` (dir is often empty; adapter still watches) |
| `~/.codex/rules/*.rules` | Tiny permission policy fragments | **Skip** ‚Äî operational config, not memory |

The session JSONL vocabulary is wider than Claude Code's. Each line
has a top-level `type` of `session_meta`, `event_msg`, `response_item`,
or `turn_context`. The semantic conversation lives inside
`response_item.payload`, which itself has five sub-types: `message`
(with roles `developer`, `user`, `assistant`), `reasoning`,
`function_call`, and `function_call_output`. Two of those carry
near-pure boilerplate that would otherwise poison the index:

- `role: "developer"` messages contain the multi-kilobyte
  `<permissions instructions>` block that prefaces every session. The
  text is identical across runs; ingesting it once per session would
  swamp BM25 and waste embedding budget.
- `role: "user"` messages whose body starts with `<environment_context>`
  are autogenerated `cwd` / `shell` / `current_date` stubs, not user
  input.

## Decision

Add `src/memstem/adapters/codex.py` with a single `CodexAdapter` class
that watches three roots ‚Äî `sessions`, `skills`, `memories` ‚Äî under a
configurable `codex_home` (default `~/.codex`). The adapter follows
the conventions established by `claude_code.py` and `openclaw.py`:

- `reconcile(paths)` walks every relevant file once at daemon start.
  `paths` is accepted for `Adapter` ABC compatibility but ignored ‚Äî the
  adapter discovers its own roots from constructor arguments.
- `watch(paths)` runs a `watchdog.Observer` against each configured
  root and re-emits the affected record on file create/modify/move.
  Full-file re-emit (no incremental offsets) for parity with
  `claude_code.py`; the downstream pipeline upserts by `ref` so
  re-emits are idempotent.
- File-suffix dispatch in the watch loop: `.jsonl` files re-parse as
  sessions, `SKILL.md` files re-parse as skills, other `.md` files
  under `memories/` re-parse as memories.

Three record shapes:

1. **`type: session`** ‚Äî one `MemoryRecord` per JSONL file. Body is the
   chronological transcript built from `response_item.payload.type ==
   "message"` entries. Filter rules:
   - `role: "developer"` ‚Üí drop entire message
   - `role: "user"` + body starts with `<environment_context>` ‚Üí drop
   - `function_call` ‚Üí summarized as `[function_call: <name>]`
   - `function_call_output` ‚Üí summarized as `[function_call_output]`
   - `reasoning` ‚Üí drop (encrypted in current rollouts; no recall value)
   - everything else ‚Üí drop
   Metadata carries `session_id` (from `session_meta.payload.id`),
   `cwd`, `turn_count`, `cli_version`, `model_provider`, and ISO
   `created` / `updated` timestamps. Tags: a project tag derived from
   the slugified `cwd` (e.g. `/home/ubuntu/memstem` ‚Üí `home-ubuntu-memstem`)
   so Codex and Claude Code sessions for the same project group
   together in search.

2. **`type: skill`** ‚Äî one record per `SKILL.md` under `skills/`,
   excluding any path component beginning with `.` (drops `.system/`
   and any future hidden dirs). Body is the markdown content; title
   pulled from frontmatter `name`, then `title`, then the first H1,
   then the parent directory name.

3. **`type: memory`** ‚Äî one record per `*.md` directly under `memories/`.
   Same parsing path as skills: frontmatter + body.

`.system/` exclusion is a hard rule, not configurable. Vendor-shipped
skills are the same on every install and don't belong in a personal
memory index. If a user actually wants them surfaced, they can copy
the file out of `.system/` themselves ‚Äî at which point it's a
user-authored skill.

A new `CodexAdapterConfig` block in `src/memstem/config.py` exposes:

```python
class CodexAdapterConfig(BaseModel):
    codex_home: Path | None = None              # defaults to ~/.codex
    sessions_root: Path | None = None           # defaults to <codex_home>/sessions
    skills_root: Path | None = None             # defaults to <codex_home>/skills
    memories_root: Path | None = None           # defaults to <codex_home>/memories
    ingest_sessions: bool = True
    ingest_skills: bool = True
    ingest_memories: bool = True
```

Each root is optional; missing roots are silently skipped. This makes
the adapter safe to enable by default ‚Äî a user without Codex installed
sees zero records emitted rather than an error.

`src/memstem/cli.py` gains a `_build_codex_adapter(cfg)` helper and
the daemon's `_run_daemon` is extended with `codex_adapter` /
`codex_paths` parameters, wired into both reconcile and watch task
lists alongside the existing adapters.

The companion piece ‚Äî telling **Codex itself** to retrieve from
Memstem ‚Äî is documented as templates under `clients/codex/`:

- `clients/codex/config.toml.fragment` ‚Äî an `[mcp_servers.memstem]`
  block users append to their `~/.codex/config.toml`. Uses the
  existing `memstem mcp` stdio command (registered at `cli.py:661`).
- `clients/codex/AGENTS.md.example` ‚Äî a starter `~/.codex/AGENTS.md`
  that instructs Codex to call `memstem_search` first for recall-style
  questions, in the same spirit as the global `~/.claude/CLAUDE.md`
  directive that already governs Claude Code sessions.
- `clients/codex/README.md` ‚Äî setup walkthrough.

Auto-wiring these system files via `memstem connect-clients codex`
is deliberately deferred to a follow-up PR so the adapter itself can
land cleanly. Operators apply the templates by hand for now.

## Consequences

**Positive:**

- Codex sessions accumulate in unified search without any change to
  Codex behavior. The "zero AI cooperation" property holds ‚Äî same
  reason this adapter design works against vendor churn.
- Cross-client project recall improves: searching for work in
  `/home/ubuntu/memstem` returns both the Claude Code and Codex
  sessions for that project because they share a derived project tag.
- Codex configured with the Memstem MCP server can read every
  ingested memory ‚Äî including Claude Code sessions, OpenClaw notes,
  and its own past sessions ‚Äî making it a peer of Claude Code in the
  "one brain, many clients" architecture.

**Negative / accepted trade-offs:**

- Full-file re-emit on every session change means a long-running Codex
  session will trigger re-parses of an ever-growing JSONL. Same
  behavior as the Claude Code adapter today; the incremental-offset
  optimization remains a Phase 2 item that would apply to both.
- The boilerplate filter (`developer` role drop,
  `<environment_context>` user-message drop) is encoded against the
  current Codex 0.130 rollout schema. If OpenAI restructures the
  rollout format in a future release we'll need to revisit; the parse
  is defensive (unknown shapes fall through to "skip"), so the worst
  case is silently dropped content rather than a crash.
- `.system/` skills are unreachable through Memstem. If a user
  genuinely wants `imagegen` surfaced in recall they have to author a
  thin wrapper skill themselves. Accepted: those skills are
  discoverable through Codex's own skill loader regardless.
- **Project `AGENTS.md` echo across sessions.** Codex auto-injects
  the project's `AGENTS.md` as the first user turn in every session
  for that `cwd`. Smoke-testing against real sessions on the
  reference install showed 8 sessions, each beginning with an
  identical kilobyte-scale AGENTS.md block. Not filtered at the
  adapter layer ‚Äî that would be brittle (the prefix can be edited)
  and would also drop the only signal that a particular AGENTS.md
  shaped a particular session. The downstream boot-echo dedup
  (`build_boot_echo_hashes` in `_run_daemon`) currently only
  considers OpenClaw and Claude Code paths; extending it to also
  walk the cwds named by ingested Codex sessions and hash any
  `AGENTS.md` found there is a clean follow-up.

## Alternatives considered

- **Ingest `.system/` skills with a `vendor:codex` tag.** Rejected:
  adds 5+ records per Codex install that are identical across users,
  inflating every vault for marginal recall value. A user who wants
  to find OpenAI's `imagegen` instructions can run `codex skill`
  commands or read the file directly.
- **Ingest `~/.codex/rules/*.rules`.** Rejected: these are tiny
  permission-policy fragments (one line each in the inspected
  install), generated by Codex itself rather than authored by the
  user. Not memory.
- **Auto-wire Codex's `config.toml` + `AGENTS.md` in this PR.**
  Rejected for scope reasons. The `connect-clients` flow already
  handles Claude Code and OpenClaw; extending it to a third client
  with its own discovery rules and idempotent-edit semantics warrants
  its own PR.
- **Stream from Codex's MCP `codex remote-control` interface instead
  of parsing JSONL.** Rejected: violates the "files on disk, no SDK
  coupling" architectural principle that makes Memstem immune to
  upgrade churn. Reading the same files Codex writes means a Codex
  point release can't break ingestion.
